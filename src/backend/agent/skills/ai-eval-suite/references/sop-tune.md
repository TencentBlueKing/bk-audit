# SOP 4: 调优

## 交互模式判断

| 用户表述 | 模式 | 做法 |
|---------|------|------|
| 具体修改指令："把 XX 改成 YY""删掉这个用例""加个 timeout" | 快捷模式 | 直接执行，完成后告知 |
| 目标导向/模糊："帮我调优""评估调优""修一下""通过率太低" | 完整模式 | 分析 → 建议 → 用户确认 → 执行 |
| 涉及 prompt 修改或多文件变更 | 完整模式 | 同上 |

## Step 1: 收集调优上下文

调优对象包括所有影响 AI 能力结果的因素：Prompt、MCP/工具链、业务代码逻辑、工具调用方式。

**首次调优时**必问：

1. **Prompt 在哪里？如何修改？**
   - 本地文件 → 直接改？还是由生成脚本产出（需改脚本重新生成）？
   - 第三方平台 → 给出建议，用户手动更新

**按需问**（根据失败类型）：
- 有哪些可选模型？（值错误/空输出多时）
- MCP/工具配置是否可调？（工具调用相关失败时）
- 业务代码是否可改？（根因在调用方逻辑时）

**首次收集后写入 Suite README 的"调优上下文"section，后续轮次直接复用。**

## Step 2: 诊断与决策

### 快捷模式
直接给出修复建议，无需全维诊断：
- 逐个列出失败用例
- 每个给出根因 + 建议修改
- 用户确认后执行

### 完整模式
按"诊断 → 决策"两步执行：

**Step 2a: 诊断**（内部推理，不需完整输出给用户）
- 扫描 5 个维度（Prompt/用例/模型/工具链/业务）
- 标记有问题的维度

**Step 2b: 决策**（输出给用户）
| 优先级 | 修改对象 | 具体内容 | 预期效果 | 成本 |
|--------|----------|----------|----------|------|
| 1 | Prompt | 增加 XX 规则 | 修复 N 个用例 | 低 |
| 2 | 测试用例 | 修正 T03 期望 | 修复 1 个用例 | 低 |

### 过拟合防护

调优修改 prompt 时，必须遵循以下规则避免过拟合：

#### 检查清单

每次修改 prompt 前自查：

1. **示例是否与测试用例重复？**
    - ❌ 示例 query 与测试用例 query 相同或高度相似
    - ✅ 示例体现同一类型但使用不同的具体值和表述

2. **规则是否足够抽象？**
    - ❌ 只列举从测试用例中抄来的具体映射（如"比较严重的→HIGH"）
    - ✅ 定义通用原则 + 多样化示例覆盖该类型

3. **修改产生的是"理解"还是"记忆"？**
    - 把示例中的具体值换掉后，模型是否仍能正确处理？

### 泛化性验证

修改 prompt 后、跑全量重评估之前，先手动验证泛化性：

1. 对每个修改点，构造 2-3 个**同类型但不在测试集中**的变体查询
2. 通过 provider 调用验证是否正确
3. 如果变体失败，说明修改过于具体，需要抽象化

## Step 3: 用户确认

将调优建议整理为清单，逐项确认：

```
调优计划：
1. ✏️ 修改 prompt — 增加枚举值映射表
   → 本地文件我直接改 / 需要你更新到 AI 平台
2. 🔧 修改测试用例 — 修正 T03 期望值
   → 我直接修改

是否确认执行？
```

## Step 4: 执行修改

- 能直接改的（测试用例、配置、本地 prompt、项目内代码）→ 直接修改
- 需用户操作的（AI 平台 prompt、模型切换、无权限代码）→ 给出具体步骤
- 每次修改后记录调优决策（见 Step 5）

## Step 5: 记录调优决策

在 suite README 中记录：

| 日期 | 问题 | 修改 | 效果 |
|------|------|------|------|
| YYYY-MM-DD | 枚举映射错误 | prompt 增加枚举映射表 | 相关用例通过 |
