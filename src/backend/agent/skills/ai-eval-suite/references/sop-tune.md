# SOP 4: 调优

## 调优交互模式

**分析 → 建议 → 用户确认 → 执行修改**

不要直接改代码。先给出分析和建议，等用户确认后再执行。

## Step 1: 收集调优上下文

分析完成后，向用户询问以下信息（之前未收集过的）：

### 必问

1. **Prompt 在哪里？**
   - 本地文件（如 `prompt/system_prompt.md`）→ 可以直接修改
   - 第三方 AI 平台上配置 → 只能给出修改建议，用户手动更新
   - 由脚本生成（如 `generate_prompt.py`）→ 修改生成脚本

2. **Prompt 是否由生成脚本产出？**
   - 如果是，生成脚本才是 single source of truth，修改必须同步到生成脚本
   - 直接改生成后的文件会在下次生成时被覆盖
   - 修改完生成脚本后，重新运行生成，确保产出文件与脚本一致

3. **Prompt 是否可以直接修改？**
   - 修改后是否需要重新部署或更新到平台

### 按需问（根据失败类型）

4. **有哪些可选模型？**（值错误或空输出较多时）
   - 当前用的什么模型
   - 是否可以切换到更强的模型

5. **MCP/工具配置是否可调整？**（工具调用相关失败时）
   - 工具返回的数据是否完整
   - 是否可以增加新工具

## Step 2: 制定调优策略

按优先级排序，成本低的先试：

### 优先级 1: 修 Prompt（成本最低，效果最直接）

**适用场景：** 字段缺失、值错误、格式错误

**常见修改：**

| 问题 | Prompt 修改方向 |
|------|---------------|
| 枚举值映射错误 | 在 prompt 中明确列出所有合法枚举值及其含义 |
| 字段缺失 | 在输出格式说明中强调该字段为必填 |
| 格式不稳定 | 增加输出格式示例，强调"只输出 JSON，不要包裹在代码块中" |
| 语义理解偏差 | 增加具体示例，用 few-shot 方式引导 |
| 否定语义处理差 | 增加否定语义的处理规则和示例 |

**建议格式：**

```
建议修改 prompt 的以下部分：

1. [位置] 枚举值说明
   [原因] 模型将"XX"映射为 `value_a` 而非 `value_b`
   [修改] 增加枚举值映射表，明确每个用户表述对应的系统值

2. [位置] 输出格式说明
   [原因] 部分输出被 markdown 代码块包裹
   [修改] 增加"直接输出 JSON，不要使用 ```json ``` 包裹"
```

### 优先级 2: 调测试用例（期望值不合理时）

**适用场景：** 期望值与业务实际不符

**常见修改：**
- 期望值过于严格（如要求精确时间戳匹配）
- 期望值与业务逻辑不一致（如枚举值已变更）
- 缺少必要的 vars（如策略列表）

**注意：** 调测试用例是为了修正不合理的期望，不是为了让测试通过而降低标准。

### 优先级 3: 换模型（Prompt 已优化但能力不足时）

**适用场景：** 复杂推理、多步骤任务持续失败

**操作：**
- 在 `promptfooconfig.yaml` 中添加新的 provider
- 同时保留旧 provider 做对比
- 关注延迟和成本的变化

```yaml
providers:
  - id: 'python:providers/provider.py'
    label: '当前模型'
    config:
      model: 'current-model'
  - id: 'python:providers/provider.py'
    label: '候选模型'
    config:
      model: 'candidate-model'
```

### 优先级 4: 改 MCP/工具（数据源问题时）

**适用场景：** 工具返回数据不完整或不准确

**操作：**
- 检查工具返回的数据是否覆盖测试用例需要的字段
- 考虑增加新的工具或扩展现有工具的返回字段

## Step 3: 用户确认

将调优建议整理为清单，逐项确认：

```
调优计划：

1. ✏️ 修改 prompt — 增加枚举值映射表
   → 需要你更新到 AI 平台

2. ✏️ 修改 prompt — 增加输出格式约束
   → 需要你更新到 AI 平台

3. 🔧 修改测试用例 — 修正不合理的期望值
   → 我可以直接修改

是否确认执行？有需要调整的吗？
```

## Step 4: 执行修改

用户确认后：
- 能直接修改的文件（测试用例、配置、本地 prompt）→ 直接修改
- 需要用户手动操作的（AI 平台 prompt、模型切换）→ 给出具体步骤
- 每次修改后记录调优决策

## Step 5: 记录调优决策

在 suite 的 output 目录或 README 中记录：

```
## 调优记录

| 日期 | 问题 | 修改 | 效果 |
|------|------|------|------|
| YYYY-MM-DD | 枚举映射错误 | prompt 增加枚举映射表 | 相关用例通过 |
| YYYY-MM-DD | 输出格式不稳定 | prompt 增加格式约束 | 格式校验通过率提升 |
```

完成后进入 SOP 5（重评估）。
